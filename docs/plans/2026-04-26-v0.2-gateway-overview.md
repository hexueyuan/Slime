## 各任务详细说明

### TASK-101: SQLite 基础设施 + 数据模型

**目标：** 引入 better-sqlite3，建表，封装数据访问层。

**交付物：**

- `src/main/db/` — 数据库初始化 + migration
- `src/main/db/models/` — 各表的 CRUD 操作封装
- `src/shared/types/gateway.d.ts` — 网关相关类型定义
- 单元测试覆盖所有 CRUD

**子步骤：**

1. 安装 `better-sqlite3` + `@types/better-sqlite3`
2. 创建 `src/shared/types/gateway.d.ts`（Channel, ChannelKey, Group, GroupItem, GatewayApiKey, ModelPrice, RelayLog, ModelSlot 等类型）
3. 创建 `src/main/db/database.ts`（初始化 SQLite、建表 DDL、migration 机制）
4. 创建各表的 DAO：`channelDao.ts`, `groupDao.ts`, `apiKeyDao.ts`, `priceDao.ts`, `logDao.ts`, `statsDao.ts`
5. 内置模型价格预设数据（从 octopus presets.go 移植主流模型）
6. 测试：每个 DAO 的 CRUD + 边界条件

### TASK-102: Gateway Core — 路由引擎

**目标：** 实现 Group → Channel 的路由解析。

**交付物：**

- `src/main/gateway/router.ts` — Group 缓存 + 路由解析
- `src/main/gateway/registry.ts` — ModelRegistry + Slot 解析

**子步骤：**

1. Group 缓存（内存 Map，从 DB 加载，CRUD 时刷新）
2. `resolve(model: string)` → 查 Group → 返回 GroupItem 列表
3. `resolveSlot(slot: ModelSlot)` → 查匹配 slot 的 Group → 返回 Group name
4. auto 模式占位（TASK-103 实现熔断器后才能完整实现）
5. 测试：路由命中/未命中、slot 匹配、缓存刷新

---

### TASK-103: Gateway Core — 负载均衡 + 熔断 + KeyPool

**目标：** 实现候选排序、熔断判断、Key 选择。

**交付物：**

- `src/main/gateway/balancer.ts` — 4 种均衡策略
- `src/main/gateway/circuit.ts` — 三态熔断器
- `src/main/gateway/keypool.ts` — 多 Key 选择

**子步骤：**

1. Balancer：round_robin（维护 index）、random、failover（按 priority 排序）、weighted（加权随机）
2. CircuitBreaker：Closed/Open/HalfOpen 三态，指数退避（30s → 60s → ... → 5min），按 channelId+keyId+model 粒度
3. KeyPool：选消费最少的 Key，429 退避标记
4. auto 模式完善：查询 slot 对应 Group 中熔断器健康度最高的
5. 测试：各均衡策略分布、熔断状态转换、Key 选择逻辑

---

### TASK-104: Outbound Adapters（Anthropic + OpenAI）

**目标：** 实现两个最核心的出站协议适配器。

**交付物：**

- `src/main/gateway/outbound/types.ts` — InternalRequest/InternalResponse 定义

**子步骤：**

1. 定义 InternalRequest / InternalResponse / InternalMessage 完整接口
2. 公共基础：HTTP 请求封装（支持代理、超时）、SSE 流式解析

4. OpenAIChatOutbound：InternalRequest ↔ OpenAI /v1/chat/completions 格式互转，包含 tool_call 格式映射
5. 测试：请求格式转换正确性（mock HTTP）、SSE 流解析

### TASK-106: Relay Loop — 完整转发链路

**目标：** 组装路由 + 均衡 + 熔断 + Outbound 为完整的请求转发链。

**交付物：**

- `src/main/gateway/relay.ts` — Relay 主循环

**子步骤：**

1. `relay(request: InternalRequest)` 主函数：
   - router.resolve(model) → GroupItem 列表
   - balancer.sort(items, balanceMode) → 排序后列表
   - 遍历候选：getChannel → keypool.selectKey → circuitBreaker.check → outbound.send
   - 成功：重置熔断、返回响应
   - 失败：记录失败、可能触发熔断、continue 下一个
   - 全部失败：返回错误
2. 流式和非流式两条路径
3. 请求计时（duration_ms）
4. 统计数据回调钩子（TASK-109 接入）
5. 集成测试：mock outbound，验证路由→均衡→熔断→重试的完整流程

### TASK-107: HTTP Server + Inbound（Anthropic Messages）

**目标：** 启动 fastify HTTP Server，实现 Anthropic Messages 入站。

**交付物：**

- `src/main/gateway/server.ts` — fastify 初始化 + 生命周期
- `src/main/gateway/inbound/anthropic.ts` — Anthropic Messages 入站解析
- `src/main/gateway/auth.ts` — API Key 认证中间件

**子步骤：**

1. fastify 服务初始化（127.0.0.1:{port}），start / stop 生命周期
2. 认证中间件：从 Bearer / x-api-key 提取 Key → 查 DB 校验
3. `POST /v1/messages` 路由：解析 Anthropic 请求 → InternalRequest → relay() → 转回 Anthropic 响应
4. `GET /v1/models` 路由：返回所有 Group 名称
5. SSE 流式响应：relay 的流式输出 → Anthropic SSE event 格式
6. 测试：用 curl / httpie 验证端到端

### TASK-109: 统计采集 + 分级存储

**目标：** 实现统计数据采集、buffer flush、分级聚合。

**交付物：**

- `src/main/gateway/stats.ts` — StatsCollector
- `src/main/gateway/tasks.ts` — 定时任务（聚合 + 清理）

**子步骤：**

1. StatsCollector：内存 buffer，`record(log: RelayLogData)` 入队
2. flush 定时器：每 30s batch INSERT 到 relay_logs
3. 小时聚合任务：relay_logs 7天前 → stats_hourly，删原始行
4. 天聚合任务：stats_hourly 30天前 → stats_daily，删小时行
5. 过期清理任务：stats_daily 90天前删除
6. 应用退出 flush 钩子
7. 费用计算：token × price / 1M
8. 接入 relay.ts 的统计回调钩子
9. 测试：buffer flush、聚合正确性、边界日期
