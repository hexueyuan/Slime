# Slime v0.2 — 内置 LLM 网关设计规格

## 概述

Slime v0.2 的核心特性：在 Electron 主进程中内建一个完整的 LLM 网关，作为所有大模型调用的统一入口。参考 octopus（Go 实现的个人 LLM 网关）的核心逻辑，用 TypeScript 重写并深度集成到 Slime 架构中。

### 核心价值

- 多渠道聚合：统一管理 Anthropic / OpenAI / Gemini / DeepSeek / Volcengine 等多个 LLM 提供商
- 双入口：内部（AgentPresenter via localhost）+ 外部（Claude Code / Codex via HTTP）
- 智能路由：负载均衡、熔断器、多 Key 轮转、会话粘性
- 协议转换：入站支持 Anthropic Messages + OpenAI Responses，出站适配各 Provider
- 数据统计：Token 消耗、费用、请求量，分级存储
- 模型抽象：Slot 系统按能力和级别选模型，支持 auto 模式

### 架构图

见 `docs/slime-v0.2-architecture.html`

## 模块 1: Gateway Core — Route + Relay

Gateway Core 是网关的核心引擎，负责路由决策和请求转发。包含路由、负载均衡、熔断、Key 管理、统计采集和 Outbound 适配器。

### 数据模型

```typescript
interface Channel {
  id: number;
  name: string;
  type: "anthropic" | "openai" | "gemini" | "deepseek" | "volcengine" | "custom";
  baseUrls: string[]; // 多端点
  enabled: boolean;
  priority: number; // 故障转移时的优先级
  weight: number; // 加权分配时的权重
  proxy?: string; // HTTP/SOCKS5 代理
  timeout?: number; // 首 Token 超时 (ms)
  createdAt: string;
  updatedAt: string;
}

interface ChannelKey {
  id: number;
  channelId: number;
  key: string;
  enabled: boolean;
  createdAt: string;
}

interface Group {
  id: number;
  name: string; // 对外暴露的模型名
  balanceMode: "round_robin" | "random" | "failover" | "weighted";
  slot?: ModelSlot; // 可选：标记能力定位
  createdAt: string;
  updatedAt: string;
}

interface GroupItem {
  id: number;
  groupId: number;
  channelId: number;
  modelName: string; // 该渠道上的实际模型名
  priority: number;
  weight: number;
}

interface GatewayApiKey {
  id: number;
  name: string;
  key: string; // sk-slime-xxx 格式
  enabled: boolean;
  expiresAt?: string;
  maxCost?: number;
  allowedModels?: string[]; // JSON 序列化存储
  createdAt: string;
}

interface ModelPrice {
  id: number;
  modelName: string;
  inputPrice: number; // $/1M tokens
  outputPrice: number;
  cacheReadPrice: number;
  cacheWritePrice: number;
  source: "preset" | "manual";
  updatedAt: string;
}
```

### Model Slot 系统

```typescript
type ModelCategory = "text" | "image";
type TextModelTier = "chat" | "reasoning";
type ReasoningLevel = "lite" | "pro" | "max" | "auto";

interface ModelSlot {
  category: ModelCategory;
  tier?: TextModelTier;
  level?: ReasoningLevel;
}
```

Slot 解析流程：

1. `resolveSlot(slot)` → 查找匹配 slot 的 Group
2. `auto` 级别：从该 tier 下所有 Group 中，选熔断器健康度最高的 Group
3. 返回 Group name 作为模型名传给网关

### 路由流程

1. 请求中的 `model` 参数 → 查 Group 缓存（内存 Map，Group CRUD 时刷新）
2. Group.balanceMode → Balancer 排序候选 GroupItem 列表
3. Relay Loop 遍历候选：
   a. 获取 Channel，检查 enabled
   b. 选 Key（选消费最少的，跳过 429 状态的）
   c. 熔断器检查（跳过已熔断的 channel+key+model 组合）
   d. 选 Outbound Adapter（根据 channel.type）
   e. 转发请求到上游 Provider
   f. 成功 → 记录统计，重置熔断器；失败 → 记录失败，可能触发熔断，continue 下一个

### 负载均衡

4 种模式（参考 octopus `balancer.go`）：

- `round_robin`：轮询，维护每个 Group 的 index
- `random`：随机打散
- `failover`：按 priority 排序，优先用高优先级渠道
- `weighted`：按 weight 加权随机

### 熔断器

三态模型（参考 octopus `circuit.go`），按 `channelId + keyId + modelName` 粒度：

- **Closed**（正常）：失败次数 < 阈值
- **Open**（熔断）：失败次数 ≥ 阈值，冷却期内直接跳过
- **HalfOpen**（探测）：冷却期过后，允许一次试探请求

指数退避冷却：初始 30s，每次 ×2，最大 5min。可配置。

### Key Pool

每个 Channel 可有多个 Key：

- 选 Key 策略：选累计消费（cost）最低的
- 429 退避：收到 429 的 Key 标记冷却时间，期间不选它
- Key 消费统计在内存中维护，定期随 stats 刷盘

### Outbound Adapters

每个 adapter 负责：将 InternalRequest 转为 Provider 的 HTTP 格式，发送请求，将响应转回 InternalResponse。

| Adapter            | 目标 API                                    | 说明                    |
| ------------------ | ------------------------------------------- | ----------------------- |
| AnthropicOutbound  | POST /v1/messages                           | 直接映射                |
| OpenAIChatOutbound | POST /v1/chat/completions                   | 消息格式转换            |
| GeminiOutbound     | POST /v1beta/models/{model}:generateContent | 格式差异大              |
| DeepSeekOutbound   | POST /v1/chat/completions                   | 同 OpenAI，baseUrl 不同 |
| VolcengineOutbound | POST /v1/chat/completions                   | 同 OpenAI，baseUrl 不同 |
| CustomOutbound     | 可配置                                      | 默认按 OpenAI 格式      |

流式响应：所有 adapter 都支持 SSE 流式，解析上游 SSE 事件，转换为统一的 InternalResponse 流。

## 模块 2: HTTP Server — 协议转换层

Slime 主进程启动一个 HTTP Server，对外提供 LLM 代理 API。

### 技术选型

`fastify`：轻量、高性能、TypeScript 友好、streaming 支持好。

### 监听配置

- 地址：`127.0.0.1:{port}`，仅本地访问
- 端口：可配置，默认 `8930`
- 生命周期：随 Slime 启动/关闭

### 入站路由（Inbound）

| 方法 | 路径            | 协议               | 典型客户端                |
| ---- | --------------- | ------------------ | ------------------------- |
| POST | `/v1/messages`  | Anthropic Messages | Claude Code, Slime 内部   |
| POST | `/v1/responses` | OpenAI Responses   | Codex                     |
| GET  | `/v1/models`    | OpenAI 风格        | 通用，返回所有 Group 名称 |

### 统一内部格式

```typescript
interface InternalRequest {
  model: string;
  messages: InternalMessage[];
  stream: boolean;
  maxTokens?: number;
  temperature?: number;
  tools?: InternalTool[];
  systemPrompt?: string;
  // 原始请求头保留，用于透传
  rawHeaders?: Record<string, string>;
}

interface InternalMessage {
  role: "system" | "user" | "assistant" | "tool";
  content: InternalContent[]; // text / image / tool_call / tool_result
}

interface InternalResponse {
  content: InternalMessage;
  usage: {
    inputTokens: number;
    outputTokens: number;
    cacheReadTokens?: number;
    cacheWriteTokens?: number;
  };
  model: string;
  stopReason: string;
}
```

每个 Inbound 将客户端请求解析为 InternalRequest，传给 Gateway Core。Core 处理完后返回 InternalResponse，Inbound 再转为客户端协议格式返回。

### 认证中间件

- `/v1/*` 路由添加认证中间件
- 支持两种传递方式：`Authorization: Bearer sk-slime-xxx` 或 `x-api-key: sk-slime-xxx`
- 校验：Key 存在 → enabled → 未过期 → 费用未超限 → 模型在白名单内
- Slime 内部使用一个自动生成的 internal key（不过期、无限额、不可删除）

### 流式处理

- stream=true：入站 Inbound 解析请求 → Core relay → Outbound 向上游发流式请求 → 上游 SSE 事件 → 转换为入站协议的 SSE 格式 → 流式返回客户端
- stream=false：Outbound 等待完整响应 → 转换 → 返回

### AgentPresenter 改造

```typescript
// 删除: ai.provider / ai.apiKey / ai.model / ai.baseUrl 配置
// 删除: createAnthropic/createOpenAI 分支
// 删除: verifyApiKey()

// 新增: 统一走 localhost 网关
private createModel(model: string) {
  const provider = createAnthropic({
    apiKey: this.gatewayPresenter.getInternalKey(),
    baseURL: `http://127.0.0.1:${this.gatewayPresenter.getPort()}`,
  })
  return provider(model)
}

// 按 slot 解析模型
const model = await this.gatewayPresenter.resolveSlot({
  category: 'text', tier: 'reasoning', level: 'auto'  // 进化默认 auto
})
```

## 模块 3: 数据层 — SQLite

### 技术选型

`better-sqlite3`：同步 API，Electron 原生模块支持好，无需异步协调。

### 数据库文件

`{userData}/.slime/gateway.db`

### 表结构

```sql
-- 渠道
CREATE TABLE channels (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  type TEXT NOT NULL,
  base_urls TEXT NOT NULL DEFAULT '[]',  -- JSON array
  enabled INTEGER NOT NULL DEFAULT 1,
  priority INTEGER NOT NULL DEFAULT 0,
  weight INTEGER NOT NULL DEFAULT 1,
  proxy TEXT,
  timeout INTEGER,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 渠道密钥
CREATE TABLE channel_keys (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  channel_id INTEGER NOT NULL REFERENCES channels(id) ON DELETE CASCADE,
  key TEXT NOT NULL,
  enabled INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 分组
CREATE TABLE groups (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL UNIQUE,
  balance_mode TEXT NOT NULL DEFAULT 'failover',
  slot_category TEXT,
  slot_tier TEXT,
  slot_level TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 分组条目
CREATE TABLE group_items (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  group_id INTEGER NOT NULL REFERENCES groups(id) ON DELETE CASCADE,
  channel_id INTEGER NOT NULL REFERENCES channels(id) ON DELETE CASCADE,
  model_name TEXT NOT NULL,
  priority INTEGER NOT NULL DEFAULT 0,
  weight INTEGER NOT NULL DEFAULT 1
);

-- 网关 API Key
CREATE TABLE api_keys (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  key TEXT NOT NULL UNIQUE,
  enabled INTEGER NOT NULL DEFAULT 1,
  is_internal INTEGER NOT NULL DEFAULT 0,
  expires_at TEXT,
  max_cost REAL,
  allowed_models TEXT,  -- JSON array or null
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 模型价格
CREATE TABLE model_prices (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  model_name TEXT NOT NULL UNIQUE,
  input_price REAL NOT NULL DEFAULT 0,
  output_price REAL NOT NULL DEFAULT 0,
  cache_read_price REAL NOT NULL DEFAULT 0,
  cache_write_price REAL NOT NULL DEFAULT 0,
  source TEXT NOT NULL DEFAULT 'preset',
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Tier 1: 原始请求日志（保留 7 天）
CREATE TABLE relay_logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  api_key_id INTEGER,
  group_name TEXT NOT NULL,
  channel_id INTEGER,
  channel_name TEXT,
  model_name TEXT NOT NULL,
  input_tokens INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL DEFAULT 0,
  cache_read_tokens INTEGER NOT NULL DEFAULT 0,
  cache_write_tokens INTEGER NOT NULL DEFAULT 0,
  cost REAL NOT NULL DEFAULT 0,
  duration_ms INTEGER NOT NULL DEFAULT 0,
  status TEXT NOT NULL,  -- 'success' | 'error'
  error TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_relay_logs_created_at ON relay_logs(created_at);

-- Tier 2: 小时聚合（保留 30 天）
CREATE TABLE stats_hourly (
  date TEXT NOT NULL,
  hour INTEGER NOT NULL,
  model_name TEXT NOT NULL,
  channel_id INTEGER NOT NULL,
  requests INTEGER NOT NULL DEFAULT 0,
  input_tokens INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL DEFAULT 0,
  cache_read_tokens INTEGER NOT NULL DEFAULT 0,
  cache_write_tokens INTEGER NOT NULL DEFAULT 0,
  cost REAL NOT NULL DEFAULT 0,
  PRIMARY KEY (date, hour, model_name, channel_id)
);

-- Tier 3: 天聚合（保留 90 天）
CREATE TABLE stats_daily (
  date TEXT NOT NULL,
  model_name TEXT NOT NULL,
  channel_id INTEGER NOT NULL,
  requests INTEGER NOT NULL DEFAULT 0,
  input_tokens INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL DEFAULT 0,
  cache_read_tokens INTEGER NOT NULL DEFAULT 0,
  cache_write_tokens INTEGER NOT NULL DEFAULT 0,
  cost REAL NOT NULL DEFAULT 0,
  PRIMARY KEY (date, model_name, channel_id)
);
```

### 统计采集流程

```
请求完成 → 内存 buffer 累积
  → 每 30s: buffer batch INSERT → relay_logs
  → 每小时: relay_logs 7天前数据 → stats_hourly，删原始行
  → 每天: stats_hourly 30天前数据 → stats_daily，删小时行
  → 每天: 删 stats_daily 90天前数据
  → 应用退出前: 强制 flush buffer
```

### 费用计算

```
cost = (inputTokens × inputPrice + outputTokens × outputPrice
      + cacheReadTokens × cacheReadPrice + cacheWriteTokens × cacheWritePrice) / 1,000,000
```

价格优先级：model_prices 表中 source='manual' > source='preset'。预设价格从 octopus `presets.go` 移植。

### 存储估算

日均 ~1000 次请求场景，稳态 2-6 MB。

## 模块 4: 管理 UI

### 侧边栏入口

Slime 左侧边栏新增"网关"入口，与进化中心同级。

### 网关面板布局

单页多区域，紧凑布局：

```
┌──────────────────────────────────────────────┐
│  今日  7天  30天                               │
│ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ │
│ │请求 924│ │Token 2M│ │费用 $74│ │缓存 68%│ │  ← 指标卡片行
│ └────────┘ └────────┘ └────────┘ └────────┘ │
├──────────────────────────────────────────────┤
│ [渠道] [分组] [接入] [日志]         ← Tab 切换│
│                                              │
│  (当前 Tab 内容)                              │
└──────────────────────────────────────────────┘
```

#### 指标卡片行

始终可见，时间范围切换（今日/7天/30天）：

- 请求次数
- Token 消耗（输入 + 输出）
- 总费用
- 缓存命中率（cacheReadTokens / inputTokens）

#### Tab: 渠道

列表：名称 | 类型 | Key 数量 | 模型数量 | 状态（正常/熔断） | 操作（编辑/删除）

编辑/新增（抽屉或内联展开）：

- 类型下拉
- Base URLs（多个，回车添加）
- Keys（多个，脱敏显示）
- 支持模型列表（手动 + 从上游拉取按钮）
- 优先级、权重、代理、超时
- 测试连接按钮

#### Tab: 分组

列表：名称 | Slot 标记 | 均衡模式 | 关联渠道数 | 操作

编辑/新增：

- 名称（= 对外模型名）
- Slot（category / tier / level，可选）
- 均衡模式下拉
- 关联渠道列表（channel + model + priority + weight）

#### Tab: 接入

列表：名称 | Key（脱敏） | 状态 | 过期时间 | 已用/上限 | 操作

新增时自动生成 `sk-slime-xxx`，配置名称、过期、费用上限、模型白名单。复制 Key 按钮。内置 Key 带标记，不可删除。

#### Tab: 日志

最近 N 条 relay_log，表格展示：时间 | 模型 | 渠道 | Token(in/out) | 费用 | 耗时 | 状态。可展开看详情。

### 全局设置面板（SettingsDialog）

网关配置 Tab（替代原 "LLM Provider"）：

- 监听端口
- 熔断器参数（失败阈值、初始冷却时间、最大冷却时间）
- 数据保留策略（7d/30d/90d 各级阈值）
- Slot 映射配置（各 slot 绑定哪个 Group）

## 模块 5: AgentPresenter 改造

### 删除

- `ai.provider` / `ai.apiKey` / `ai.model` / `ai.baseUrl` 配置项
- `createModel()` 中的 provider 分支逻辑
- `verifyApiKey()` 方法

### 新增

- `gatewayPresenter.getInternalKey()` 获取内部 Key
- `gatewayPresenter.getPort()` 获取网关端口
- `gatewayPresenter.resolveSlot(slot)` 解析 Slot 为模型名
- 统一使用 `createAnthropic` 连接 `localhost:{port}`
- 进化场景默认 `{ category: 'text', tier: 'reasoning', level: 'auto' }`

### auto 模式

从该 tier 下所有 Group 中，选熔断器健康度最高（最近失败率最低）的 Group。多个都健康则按 priority 选。

## 模块 6: Onboarding 改造

### 新流程（4 步）

**Step 1: Welcome**

- Slime 自我进化概念介绍
- 说明内置 LLM 网关

**Step 2: AddChannel**

- 类型下拉（Anthropic / OpenAI / Gemini / DeepSeek / Volcengine / Custom）
- Base URL（按类型自动填充默认值）
- API Key 输入
- "测试连接"按钮 → 验证连通性 + 拉取模型列表
- 成功后展示模型列表，勾选要启用的
- 自动为勾选模型创建同名 Group

**Step 3: SlotMapping**

- 基于 Step 2 勾选的模型，分配 Slot：
  - reasoning auto / lite / pro / max
  - chat
- 可跳过（后续在设置中配置）
- 智能默认：只有一个模型时全部 slot 指向它

**Step 4: Identity + Complete**

- 用户名输入
- 汇总展示：渠道名、模型数量、Slot 映射
- 完成引导
