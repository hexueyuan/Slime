一键整理文档、更新记忆、提交并推送代码。

执行以下步骤：

## 第一步：了解变更

1. 运行 `git status` 查看当前未提交变更
2. 运行 `git log @{push}..HEAD --oneline` 查看已提交但未 push 的 commits（如果没有 push 跟踪分支则用 `git log origin/main..HEAD --oneline`）
3. 运行 `git diff` 和 `git diff --cached` 查看未暂存和已暂存的具体变更
4. 综合以上信息，梳理出本次 ship 涵盖的所有变更（未提交 + 已提交未push）

## 第二步：更新 docs 文档

基于第一步收集的变更内容，**逐一读取**以下文档并判断是否需要同步更新：

### 必查清单（每个文件都必须 Read 后再决定是否修改）

1. **CLAUDE.md**（项目根目录）：读取，检查 Presenter 表格、工具数量、项目结构等是否与代码一致
2. **docs/AGENTS.md**：读取，检查内容是否与 CLAUDE.md 保持同步（两者共享架构描述）
3. **specs/ 和 plans/**：如果变更涉及某个 TASK 的实现，读取对应 plan，更新 checkbox 进度（`- [ ]` → `- [x]`）

### 更新原则

- CLAUDE.md 和 docs/AGENTS.md 内容必须保持一致，改一个必须检查另一个
- 如果变更涉及架构调整（新增模块、Presenter、工具等），两个文件都要同步
- 新 plan/spec：如果本次变更是一个新任务但还没有对应文档，不需要创建
- 如果没有需要更新的文档，跳过本步

## 第三步：更新记忆

读取 MEMORY.md（路径：`~/.claude/projects/-Users-hexueyuan-Workroot-src-github-com-hexueyuan-Slime/memory/MEMORY.md`），基于本次变更：

### 添加

- 本次变更中遇到的重要 bug 修复经验、架构决策、技术陷阱
- 任务进度更新（如有新 TASK 完成或推进）
- 新发现的关键模式或约定

### 清理

- 已不再适用的信息（如已实现的 stub 模块标注、已过时的技术方案）
- 过于细节的临时信息（只保留对未来有价值的经验）
- 与当前代码状态矛盾的描述

### 原则

- 保持 MEMORY.md 在 200 行以内
- 信息按语义组织，不按时间堆积
- 只记录跨会话有价值的信息

## 第四步：提交代码

1. 运行 `git status` 确认所有第二步、第三步产生的变更都已识别（防止遗漏文件）
2. 将所有变更添加到暂存区（排除 .env 等敏感文件）
3. 运行 `git diff --cached --stat` 检查暂存内容，确认无遗漏
4. 根据全部变更内容生成符合 Conventional Commits 规范的提交信息
5. 提交变更
6. 如果第一步就没有任何变更（未提交 + 未push），不执行提交

## 第五步：推送到远端

1. 确认当前分支有远程跟踪分支
2. 运行 `git push` 推送所有本地 commits
3. 如果推送失败，报告错误但不强制推送

## 注意事项

- 提交信息格式：`type(scope): subject`，单行，不用 HEREDOC
- 如果本次有多类变更（代码 + 文档 + 记忆），可以拆分为多个 commit（代码改动一个、文档更新一个），也可以合并为一个，视变更规模而定
- 如果没有任何变更则不执行任何操作
- 每步完成后简要告知用户该步结果
